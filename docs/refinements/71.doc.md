# Refinement: Issue #71 — Automate Documentation

## Summary

Add a nightly `doc-maintainer` job (scheduled at 1 AM) that automatically generates and maintains documentation under `docs/` in each repository that claws actively works with. The main entry point is `docs/OVERVIEW.md`, but Claude is free to create additional dedicated documents for detailed subjects (e.g., `docs/database-schema.md`, `docs/api-design.md`). The job detects changes by comparing HEAD against the last `[doc-maintainer]` commit in git history, then uses Claude to generate or refresh documentation based on the current state of the codebase. It creates a PR labeled `claws-mergeable` for automatic merging once CI passes. The documentation is optimized for providing AI context when planning and implementing new work — giving Claude background on repo architecture, patterns, and key concepts.

Additionally, a new `auto-merger` job monitors all PRs labeled `claws-mergeable` across all repos and merges them (squash) once CI is green. This is a generally useful capability — doc-maintainer PRs, Dependabot PRs, and any other PRs can be auto-merged by applying the label.

Once docs exist, the issue-planner and issue-worker prompts are updated to instruct Claude to read `docs/OVERVIEW.md` (and any linked docs) before starting work.

  - Returns the current HEAD SHA

### 5. `src/jobs/doc-maintainer.ts` — New file (core job)

This is the main new file. It follows the same structure as `issue-worker.ts` and `issue-planner.ts`:
- Exports `run(repos: Repo[]): Promise<void>`
- Iterates repos with per-repo error handling via `reportError()`
- Uses `claude.enqueue()` for serial Claude invocations
- Creates/removes worktrees in try/finally

**Flow for each repo:**

#### Step 0: Skip repos claws isn't working with

Check if a local clone exists at `~/.claws/repos/{owner}/{name}`. If not, skip the repo — claws only maintains docs for repos it has already checked out and worked with. This avoids generating documentation for repos that claws has never touched.

```typescript
const repoDir = path.join(CLAWS_DIR, "repos", repo.owner, repo.name);
if (!fs.existsSync(repoDir)) continue;
```

#### Step 1: Check for existing open docs PR

Use `gh.listPRs(repo.fullName)` and filter for PRs where `pr.headRefName.startsWith("claws/docs-")`. If an open docs PR exists, skip this repo to avoid duplicate work.

#### Step 2: Check if maintenance is needed

- Create a worktree from the default branch: `claude.createWorktree(repo, branchName)` where `branchName = "claws/docs-${claude.randomSuffix()}"`
- Get the current HEAD SHA via `claude.getHeadSha(wtPath)`
- Look up the last doc-maintainer commit via `claude.getLastDocMaintainerSha(wtPath)`
- If no previous doc-maintainer commit → first run, docs needed, proceed to step 3
- If previous commit SHA equals HEAD SHA → no changes since last update, skip and clean up worktree
- Otherwise → code has changed since last doc update, proceed to step 3

#### Step 3: Generate/update documentation

Run Claude in the worktree with the doc generation prompt. The prompt works with the current state of the repository rather than examining individual commits — this is efficient whether it's the first run or updating after many changes.

```
You are maintaining documentation for the repository {fullName}.

Your goal is to create or update documentation under `docs/` that is
optimized for providing context when planning and implementing new features
and bug fixes.

Steps:
1. Run `mkdir -p docs` to ensure the directory exists.
2. Read the codebase to understand its current structure, purpose, and key
   patterns.
3. If `docs/OVERVIEW.md` exists, read it and all docs it links to, then
   update them to reflect the current state of the code. Preserve accurate
   content and update anything outdated. If it doesn't exist, create it
   from scratch.
4. `docs/OVERVIEW.md` is the main entry point and should include:
   - **Purpose**: What this repo does and its role (2-3 sentences)
   - **Architecture**: Key directories, modules, and how they fit together
   - **Key Patterns**: Important conventions, data flow, and design decisions
   - **Configuration**: Key config values and environment variables
5. For complex subsystems that need detailed coverage, create dedicated
   documents (e.g., `docs/database-schema.md`, `docs/api-design.md`) and
   link to them from OVERVIEW.md. Keep each focused on one subject.
6. Keep OVERVIEW.md concise (200-500 lines). Dedicated docs can be longer
   as needed for thorough coverage.
7. Commit with message: "docs: update documentation [doc-maintainer]"

Do NOT make any code changes. Only update documentation.
```

The `[doc-maintainer]` marker in the commit message allows future runs to find when docs were last updated by searching git history.

#### Step 4: Push and create PR

- Check `claude.hasNewCommits(wtPath, repo.defaultBranch)`
- If no new commits → log warning, skip
- If commits exist:
  - `claude.pushBranch(wtPath, branchName)`
  - Create PR with title `docs: update documentation for {repo.name}` and body explaining it's an automated docs update
  - Apply `claws-mergeable` label to the PR (the auto-merger job will handle merging once CI passes)

#### Step 5: Cleanup

Remove the worktree in a `finally` block, following the existing pattern from all other jobs.

**Task tracking:** Unlike the label-driven jobs, doc-maintainer uses `item_number: 0` for its task records since there's no issue associated. The task is recorded for crash recovery and orphan cleanup.

### 6. `src/scheduler.ts` — Add scheduled job support

Extend the `Job` interface to support time-based scheduling as an alternative to interval-based. Add an optional `scheduledHour` field. When present, the job is not run immediately on startup — instead it waits until the next occurrence of that hour, then repeats every 24 hours.

```typescript
export interface Job {
  name: string;
  intervalMs: number;
  scheduledHour?: number;  // 0-23, run at this hour daily instead of on interval
  run: () => Promise<void>;
}
```

In `startJobs()`, when `scheduledHour` is defined, calculate the delay until the next occurrence of that hour and use `setTimeout` → `setInterval(24h)` instead of running immediately:

```typescript
function msUntilHour(hour: number): number {
  const now = new Date();
  const target = new Date(now);
  target.setHours(hour, 0, 0, 0);
  if (target <= now) target.setDate(target.getDate() + 1);
  return target.getTime() - now.getTime();
}

// In the job loop:
if (job.scheduledHour !== undefined) {
  const delay = msUntilHour(job.scheduledHour);
  log.info(`Scheduling ${job.name} for ${job.scheduledHour}:00 (in ${Math.round(delay / 60000)} min)`);
  timers.push(setTimeout(() => {
    tick();
    timers.push(setInterval(tick, 24 * 60 * 60 * 1000));
  }, delay));
} else {
  tick();
  timers.push(setInterval(tick, job.intervalMs));
}
```

### 7. `src/jobs/auto-merger.ts` — New file (general-purpose auto-merge job)

A new job that merges any PR labeled `claws-mergeable` once CI is green. This is not specific to doc-maintainer — it provides a general-purpose auto-merge capability that can be used by any workflow (doc-maintainer PRs, Dependabot PRs, or any PR a human labels).

The job runs on a regular interval (e.g., every 10 minutes), following the same pattern as `ci-fixer.ts`.

**Flow for each repo:**

1. List all open PRs via `gh.listPRs(repo.fullName)`
2. Filter for PRs with the `claws-mergeable` label
3. For each matching PR:
   - Check the PR's mergeable state and CI status via `gh pr view --json mergeStateStatus,statusCheckRollup`
   - If all checks pass and the PR is mergeable → call `gh.mergePR(repo.fullName, prNumber)` and remove the `claws-mergeable` label
   - If checks are still pending → skip (will be picked up on next cycle)
   - If checks have failed → log a warning, leave the label for manual attention

**Why a separate job instead of inline merge?**

Calling `gh pr merge --auto` directly works when branch protection requires checks. But having a dedicated job gives claws visibility into what's pending and lets any workflow participate by applying the label — no code changes needed per consumer.

## Potential Risks and Edge Cases

### 1. First-run storm

~~On first deployment, every repo will have no `[doc-maintainer]` commit, so all repos will need docs generated. With ~200 repos across both owners, this could queue many Claude invocations.~~

**Resolved:** The doc-maintainer only processes repos that claws has already cloned locally (i.e., repos with an existing directory under `~/.claws/repos/{owner}/{name}`). This naturally limits scope to repos that claws is actively working with, avoiding unnecessary documentation generation for repos claws has never touched.

### 2. Claude queue saturation

Each repo that needs docs updated requires a full doc generation Claude call, which could block the serial queue for extended periods.

**Mitigation:** The job runs at 1 AM when no other activity is expected. Most repos will have no new commits since their last `[doc-maintainer]` commit and will be skipped before reaching Claude (the HEAD vs last-marker-SHA comparison is a cheap git operation).

### 3. Duplicate PRs

If a docs PR is created but not merged before the next nightly run, we'd create a duplicate.

**Mitigation:** The `listPRs` check at the top of each repo's processing filters for open PRs with `claws/docs-*` branch prefix and skips the repo if one exists.

### 4. `--head` flag limitation

The `gh pr list --head` flag requires an exact branch name, not a prefix. Since doc branches include a random suffix (`claws/docs-ab12`), we can't use `--head` for prefix matching.

**Mitigation:** Use `gh.listPRs()` (list all open PRs) and filter in code for `pr.headRefName.startsWith("claws/docs-")`. This is slightly more expensive than a filtered query but correct. For repos with many open PRs, this is still fast since we're just filtering JSON.

### 5. Large repos producing oversized docs

Very large repos could produce sprawling documentation beyond the 200-500 line target.

**Mitigation:** The prompt explicitly targets 200-500 lines. For the repos under `stjohnb` and `St-John-Software`, this shouldn't be an issue. If it becomes one, the prompt can be tuned.

### 6. Repos with no meaningful code

~~Empty repos, forks, or trivially small repos will get an unnecessary Claude call on first run.~~

**Resolved:** The job only processes repos that claws has already cloned locally, which means repos where claws has done meaningful work. Combined with the `[doc-maintainer]` marker being self-correcting after the first run, this is a non-issue.

## Design Decisions

### Why git commit markers instead of a database table?

Using a `[doc-maintainer]` marker in the commit message and searching git log is simpler than maintaining a separate `doc_runs` table. The tracking is self-contained within each repository's git history, requires no database schema changes, and naturally survives database resets or migrations. The git log search (`git log --grep`) is fast and gives us exactly what we need — the SHA of the last doc update.

### Why compare current state instead of examining individual commits?

When docs are first generated or haven't been updated in a long time, examining each individual commit would be inefficient. Instead, the doc generation prompt reads the current codebase and compares it against the existing `docs/OVERVIEW.md` (if any). This approach works equally well for first runs and incremental updates, and avoids the need for a separate significance evaluation step.

### Why a scheduled time instead of an interval?

Intervals are reset by deployments, which means a 24-hour interval could run much more or less frequently depending on deployment cadence. Scheduling at a fixed hour (1 AM) ensures consistent, predictable execution regardless of when the service is deployed or restarted.

### Why a `claws-mergeable` label instead of direct auto-merge?

Rather than having doc-maintainer call `gh pr merge --auto` directly, it applies a `claws-mergeable` label and a separate `auto-merger` job handles the merge once CI passes. This decouples the merge capability from any specific job, making it reusable — Dependabot PRs, manual PRs, or any future automated PRs can be auto-merged by simply applying the label. The auto-merger job checks CI status before merging, providing a safety net.

### Why only process repos with local clones?

The doc-maintainer checks for a local clone at `~/.claws/repos/{owner}/{name}` before processing a repo. This ensures docs are only maintained for repos claws is actively working with, rather than generating docs for every repo across both GitHub owners. As claws starts working with new repos (via issue-planner, issue-worker, etc.), their clones are created automatically, and doc-maintainer will pick them up on the next nightly run.

### Why filter open PRs in code instead of adding `listPRsByHead`?

The `gh pr list --head` flag requires an exact branch name match. Since doc branches include a random suffix, we'd need to list all open PRs and filter anyway. Reusing the existing `listPRs()` function avoids adding a new GitHub helper that doesn't provide real value.

**Update based on implementation:** If prefix filtering becomes needed by other jobs in the future, `listPRsByHead` could be added then. For now, YAGNI.
