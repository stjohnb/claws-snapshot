# Potential Ideas

### Review feedback pattern extraction for per-repo guidance

Add a scheduled job (or extend doc-maintainer) that analyzes review comments left on merged Claws PRs over the past N days. Extract recurring themes — e.g. 'always add tests for new utility functions', 'avoid inline styles in this repo', 'use the existing ErrorBoundary instead of try/catch in components'. Compile these into a `docs/coding-guidance.md` (or append a section to OVERVIEW.md) that gets injected into issue-worker and improvement-identifier prompts. The data is already available: merged PRs with `claws/issue-*` branches have review comment history accessible via `gh`. Over time this creates a feedback loop where Claws learns from the operator's review preferences and produces PRs that need fewer review rounds. Track which patterns reduce review iterations to validate the approach is working.

### Task difficulty estimation from historical data

Use the SQLite task history to build a simple difficulty model for incoming work. For each completed task, the database already records duration, success/failure, and the job name. Correlate these with issue characteristics (body length, number of files mentioned, presence of images, multi-phase plans) to estimate whether a new issue is likely to be a quick fix or a complex multi-round task. Surface this estimate on the dashboard queue page as a complexity indicator (e.g. simple/moderate/complex) so the operator can prioritize review time. Could also be used to set per-task Claude timeouts — simple tasks get a shorter timeout to fail fast if Claude gets stuck, while complex tasks get extended time.

### Cross-repo improvement propagation

Extend improvement-identifier to track improvements it has successfully implemented (merged PRs with `claws/improve-*` branches). When an improvement is merged in one repo — such as adding input validation, fixing an error handling pattern, or updating a dependency usage — scan other managed repos for the same anti-pattern. If found, create a targeted improvement issue referencing the original fix as prior art. This leverages the fact that repos managed by the same developer often share similar patterns and dependencies. Implementation: after improvement-identifier's analysis phase, cross-reference each suggested improvement's description against a list of recently merged improvements from other repos. This avoids the current situation where the same category of improvement gets independently discovered (and sometimes differently resolved) across repos.

### Failed attempt context injection for retried tasks

When a task is retried on the same item and job (e.g., ci-fixer re-processing a PR after a previous failure, or issue-worker retrying after a timeout), query the task table for prior attempts on that item and inject the previous attempt's error message, duration, and — if available — the last stdout snippet from ClaudeTimeoutError diagnostics into the new prompt. Currently each Claude session starts with zero knowledge of what was tried before, so it frequently repeats the exact same failing approach. The data already exists: the tasks table records errors and the error reporter captures ClaudeTimeoutError diagnostics (lastOutput, lastStderr). Implementation: in each job's Claude invocation, call a new helper like `getPriorAttemptContext(repo, itemNumber, jobName)` that queries recent failed tasks (last 48 hours) and formats a 'Previous attempts' prompt section. Cap the injected context at ~5KB to avoid bloating prompts. This is distinct from the MCP server idea (#480) which gives Claude access to general Claws state — this specifically provides failure-specific learning context to avoid repeating mistakes.

### Cumulative review round history for review-addresser

When review-addresser processes a PR that has been through multiple review rounds, include a condensed summary of prior rounds — what feedback was given, what changes were made in response, and what the reviewer said afterward. Currently each review-addresser invocation only sees the current unaddressed comments, so Claude lacks context about the conversation arc. For example, if round 1 said 'add error handling' and Claude added a try/catch, then round 2 says 'use the existing ErrorBoundary instead', Claude needs to understand that its first approach was rejected. Implementation: when building the review-addresser prompt, fetch all PR comments with 🚀 reactions (previously addressed) and the corresponding commits, format as a 'Review history' section showing the feedback→response pairs. This provides conversation continuity without requiring Claude to re-read the entire comment thread and piece together what happened.

### Live Claude output streaming via SSE on the dashboard

Add a GET /logs/live SSE endpoint that streams Claude's stdout output in real-time for currently running tasks. The dashboard would show a 'Live' tab or button on active tasks that opens a terminal-like view of Claude's output as it happens. Currently the operator must wait until a task completes to see any output in the logs page — for tasks that run up to 6 hours, this means no visibility into whether Claude is making progress, stuck in a loop, or heading in the wrong direction. The claude.ts module already captures stdout via the spawned process; this would add a lightweight pub/sub mechanism where the spawn handler publishes chunks to registered SSE subscribers. Include auto-scroll, a pause button, and a byte counter. The heartbeat log (every 5 minutes) already tracks stdout byte count — this surfaces the actual content. Rate-limit the SSE stream (e.g., buffer and flush every 500ms) to avoid overwhelming browsers with high-throughput output. Authentication would use the same token/cookie mechanism as other dashboard endpoints.

### Queue saturation throttling for discretionary jobs

When the Claude queue depth exceeds a configurable threshold (e.g., 2× maxClaudeWorkers), automatically suppress scheduled ticks for discretionary jobs (improvement-identifier, idea-suggester, doc-maintainer) until the queue drains below the threshold. Currently all jobs fire on their timers regardless of queue depth, so a busy period with many CI fixes and review-addresser tasks can see improvement-identifier and idea-suggester adding more items to an already saturated queue. The priority queue idea (already suggested) addresses ordering within the queue; this addresses not adding to a saturated queue in the first place. Implementation: in the scheduler's tick function, before calling a job's run(), check `queueStatus()` from claude.ts. If `waiting >= threshold` and the job is in a configurable `discretionaryJobs` list, skip the tick and log a 'deferred due to queue saturation' message. The job would fire on its next scheduled tick if the queue has drained. Manual triggers via the dashboard would bypass saturation throttling.

### Time-of-day aware scheduling for Claude-intensive jobs

Add a configurable 'active hours' window (e.g., 9am–6pm weekdays) during which discretionary Claude-intensive jobs (improvement-identifier, idea-suggester) are suppressed, reserving Claude capacity for interactive work like issue-worker and ci-fixer that the operator is likely waiting on during working hours. Conversely, batch-style jobs run during off-hours when the queue is typically empty. Currently, improvement-identifier runs at 3am (good) but if manually triggered during the day or if the schedule shifts, it competes with time-sensitive tasks. This goes beyond the existing skipWeekends flag by adding intra-day awareness. Implementation: add an `activeHoursOnly` flag to the Job interface and a `quietHours` config range. Jobs marked `activeHoursOnly: false` (batch jobs) would be deferred to quiet hours. The dashboard would show a 'quiet hours' indicator and which jobs are deferred.

### Dashboard action audit log with history page

Record all mutating dashboard actions in a new audit_log SQLite table: job triggers (POST /trigger/:job), pause/resume (POST /pause/:job), skip/unskip/prioritize/deprioritize (POST /queue/*), PR merges (POST /queue/merge), config changes (POST /config), task cancellations (POST /cancel), and WhatsApp unpair. Each entry stores timestamp, action type, target (job name, repo/number, or config field), client IP (from X-Forwarded-For or socket), and the authenticated token hash (to distinguish multiple operators if auth tokens are ever per-user). Add a GET /audit dashboard page showing a filterable, paginated history. This provides accountability ('who triggered improvement-identifier at 2am and burned 3 hours of Claude time?'), debugging context ('was the job manually triggered or scheduled?'), and change tracking ('who changed maxClaudeWorkers from 2 to 4?'). Apply the same log retention policy as job_logs (configurable via logRetentionDays). No Claude invocation needed — purely SQL inserts in the existing request handlers.

### GitHub Actions version pinning scanner

Add a daily scanning job (matching the ubuntu-latest-scanner and concurrency-scanner pattern) that scans .github/workflows/*.yml files across all cloned repos for GitHub Actions references using mutable tags instead of pinned SHA commits. Detect uses: directives like actions/checkout@v4 or actions/setup-node@v3 (tag-based) versus actions/checkout@<full-sha> (pinned). Unpinned actions are a well-documented supply chain attack vector — a compromised action maintainer can push malicious code to an existing tag, which would then execute in all consumers' CI pipelines. First-party actions (actions/*, github/*) could be flagged at a lower severity than third-party actions. Create a deduped alert issue per repo (title: 'Alert: GitHub Actions using unpinned versions') with a table of unpinned action references, their workflow file and line number, and the recommended pinned SHA. Include a note about tools like pin-github-action that automate the pinning process. Does not require Claude — purely a filesystem scan with regex matching and gh CLI issue creation, exactly like the existing scanner jobs.

### Cross-job context chain for issue lifecycle

Create a persistent context chain that follows an issue through its entire lifecycle across different jobs. When the planner processes an issue, it records a structured summary (key decisions, identified challenges, architectural notes) to a new `task_context` SQLite table keyed by `{repo, item_number, job_name}`. When the implementer picks up the same issue, it receives this planner context alongside the plan text. When ci-fixer later processes the resulting PR, it receives both the planner and implementer context — e.g., 'the implementer chose to modify AuthService rather than creating a new adapter because the existing test suite depends on AuthService's interface — CI failures in auth tests likely need test updates, not implementation changes.' When review-addresser processes feedback, it has the full chain. Currently, each job builds its prompt in isolation: issue-worker gets the issue body and comments, ci-fixer gets the PR diff and failure logs, review-addresser gets the review comments. None of them know what the upstream jobs decided or why. Implementation: add a `task_context` table with columns `repo`, `item_number`, `job_name`, `context_json`, `created_at`. After each Claude task completes successfully, extract structured context from the task output (files changed, approach rationale, key decisions) via lightweight parsing of the git diff and Claude's output. Each job's prompt builder calls `getContextChain(repo, itemNumber)` to retrieve prior entries and formats them as a 'Prior context from earlier pipeline stages' prompt section. Cap total chain size at 8KB, summarizing older entries if needed. Link issue numbers to PR numbers via the existing `getOpenPRForIssue()` helper so that PR-stage jobs can look up the originating issue's context chain.

### Issue genealogy context injection for derived issues

When issue-worker processes an issue that references other issues (via `#123` syntax, 'follow-up to', 'extracted from', or created by ci-fixer's unrelated failure filing), automatically resolve the referenced issues and inject their resolution context into the prompt. For a follow-up issue, include: the parent issue's title and body summary, the approach taken in the parent's PR (from the PR description and diff stats), and any review feedback the parent received. This gives Claude critical context about why the current issue exists and what constraints the prior work established. For example, if issue #200 says 'Add error handling to the payment flow (follow-up to #185)' and #185's PR added the payment flow with a reviewer note that 'error handling was deferred to a follow-up,' Claude immediately understands the scope, existing code structure, and the reviewer's expectations. Currently, Claude would need to independently discover the payment flow code and guess at the design decisions. Implementation: in the issue-worker prompt builder, regex-match `#\d+` references in the issue body (excluding self-references), fetch their state via `gh issue view` and `gh pr list --search`, extract the linked PR's description and merge commit diff stats, and format as a 'Related issue context' prompt section. For issues created by ci-fixer's unrelated failure classification (which includes a metadata comment with the originating PR number), automatically include the originating PR's context. Limit to 3 referenced issues and 4KB total to avoid prompt bloat.

### Codebase change awareness for concurrent and recent tasks

Before a Claude task begins work on an issue, detect whether other recently completed tasks modified files that are likely relevant to the current task, and inject this awareness into the prompt. Parse the issue body and plan for file references (explicit paths like `src/auth.ts`, component names like 'AuthService', module references like 'the auth module'), then query the task table for completed tasks on the same repo in the last 48 hours. For each recent task, check which files it modified by examining its branch diff against main via `git diff --name-only origin/main...origin/<branch>`. If overlap is detected (same files referenced in the current issue were modified by a recent task), inject a warning: 'Note: `src/auth/service.ts` was modified 6 hours ago by task #42 (ci-fixer). These changes are now on main. Base your work on the current state of these files, not on any prior knowledge of their contents.' This prevents Claude from making assumptions based on stale understanding of the codebase and reduces merge conflicts when tasks process related issues in quick succession. Currently, concurrent tasks on the same repo can unknowingly create conflicting changes because neither is aware the other exists. Implementation: add a `getRecentChangedFiles(repo, hoursBack)` helper that queries completed tasks with worktree branches still reachable and runs `git diff --name-only`. Call this in the issue-worker and ci-fixer prompt builders. The file-reference extraction from issue bodies can use simple regex for paths and a fuzzy match against the repo's file listing for component names.

### Anomaly detection alerts for task duration and failure spikes

Implement lightweight statistical anomaly detection on task metrics that alerts the operator when something unusual is happening, without requiring manual threshold configuration. For each job type, maintain a rolling baseline from the last 50 completed tasks: mean duration, standard deviation, and historical failure rate. When a running task's duration exceeds 2 standard deviations above the mean for its job type, send a Slack alert: 'ci-fixer task on owner/repo #42 has been running for 3.2 hours (typical: 45±20 minutes) — may be stuck or facing an unusually complex problem.' When the failure rate over the last 5 tasks for a job exceeds 2 standard deviations above its historical rate, alert: 'issue-worker failure rate is 80% over the last 5 tasks (historical average: 15%) — investigate common cause.' This is distinct from job health scoring with circuit-breaking (#482) which uses fixed thresholds and automatically pauses jobs — this uses statistical baselines derived from actual performance data and is purely observational (alerts only, no automatic action). The value is early warning: the operator learns about anomalies as they develop, not after a job has burned through its failure threshold. Implementation: add a `getJobBaseline(jobName)` helper that queries the tasks table for the last 50 completed tasks of that job type and computes mean/stddev for duration and success rate. Check at the existing 5-minute heartbeat interval for running tasks, and after each task completion for failure rate spikes. Use the existing Slack infrastructure with a distinct message format. Include a 1-hour deduplication window to avoid alert fatigue for the same anomaly.

### Cascade trigger condition expressions

Extend the Job interface's triggers field from a simple string array of job names to support conditional trigger expressions that control when downstream jobs fire. Currently, triggers fire whenever the upstream job's run produces any tasks (taskCount > 0). Add support for conditions like: trigger only if specific outcome types occurred (e.g., pr-dispatcher triggers auto-merger only if a review was posted, not after every CI fix), trigger only if the queue for the downstream job is below a threshold (prevent piling work onto an already-busy job), and trigger with a configurable delay based on the upstream outcome (e.g., wait 60 seconds after ci-fixer pushes to let CI start before pr-dispatcher re-checks). Define conditions as objects: { job: 'auto-merger', condition: 'outcomeIncludes:reviewed', delay: 30000 }. The scheduler already implements cascade triggering with a 10-second delay and max depth of 5. This extends the expressiveness without changing the core mechanism. Implementation: change the triggers type from string[] to (string | TriggerCondition)[]; plain strings retain current behavior for backward compatibility. After a job run completes, evaluate each trigger condition against the run's task outcomes (already stored in DB). The outcomeIncludes condition checks if any task in the run has a matching prAction or job_name. The queueBelow condition queries queueStatus(). This reduces unnecessary downstream triggers — currently pr-dispatcher triggers itself after every run that produces tasks, even if none of those tasks changed merge eligibility.
