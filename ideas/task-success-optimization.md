# Task success optimization

### Claude model selection based on task complexity signals (#670)

Add automatic model selection logic that chooses between faster/cheaper and more capable Claude models based on task complexity signals. Classify incoming tasks using heuristics: (1) issue body length and structure (short single-paragraph issues → simpler model; multi-section issues with code blocks and images → more capable), (2) historical success rate for this job+repo combination (if previous attempts failed, escalate to a more capable model), (3) plan phase count (single-PR plans → simpler; multi-PR plans → more capable), (4) file count mentioned in the plan or issue. Map these signals to model tiers: simple tasks (label syncing, doc updates, straightforward bug fixes) use a faster model, while complex tasks (multi-file refactors, new features, CI investigations) use the most capable model. Implement as a `selectModel(taskSignals)` helper called before `runClaude()`. The Claude CLI supports model selection via the `--model` flag. Store the model used in the task record for analysis. Add a config override to force a specific model for all tasks if the operator prefers consistency over optimization.

### Post-task plan accuracy validation and update (#671)

After issue-worker completes a multi-phase plan's first PR, compare what was actually implemented (from the git diff) against what the plan specified for that phase. If significant deviations are detected — for example, the plan said 'add a new UserService class' but the diff shows modifications to an existing AuthService instead — automatically update the plan comment on the issue to reflect reality before proceeding to the next phase. Currently, if Claude's implementation diverges from the plan (which happens when Claude discovers the plan's approach won't work), subsequent phases still reference the original plan text, causing confusion and wasted work. Implementation: after each phase's PR is merged, invoke a lightweight Claude call that compares the phase's plan text with the actual diff and produces an updated plan. Use the existing `makePlanUpdateFooter()` machinery to edit the plan comment. This ensures phase N+1 always has an accurate picture of what phase N actually accomplished.
