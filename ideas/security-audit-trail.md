# Security & audit trail

### Prompt injection detection for issue and PR content (#666)

Before any user-supplied text (issue bodies, PR comments, review feedback) is injected into a Claude prompt, run it through a lightweight scanner that detects common prompt injection patterns. Check for: explicit instruction overrides ('ignore all previous instructions', 'you are now', 'system prompt:'), encoded payloads (base64-encoded instructions, Unicode homoglyph obfuscation), and suspicious markdown (hidden text via HTML comments containing instructions, zero-width characters). Score each piece of content; if the score exceeds a threshold, either sanitize the suspicious sections (replace with '[content redacted — potential prompt injection]') or skip the item entirely and post a Slack alert with the matched patterns. This complements the existing fork PR filtering (which blocks untrusted PR authors) by protecting against injection from trusted collaborators whose accounts may be compromised, or from issue templates that include injected content. Implement as a utility function called in each agent's prompt builder before text interpolation. False positives should be conservative — only flag clearly adversarial patterns.

### Pre-push diff safety scanning (#667)

After Claude completes a task and before `pushBranch()` sends commits to GitHub, run a configurable set of safety checks on the diff. Scan for: (1) potential secrets — regex patterns matching API keys, tokens, passwords, private keys, and connection strings (similar to gitleaks patterns), (2) unexpected file modifications — changes to .github/workflows/, deploy/, Dockerfile, or other infrastructure files when the task was scoped to application code, (3) new external network calls — additions of fetch/axios/http calls to domains not already present in the codebase, (4) large binary files accidentally committed. If any check fails, skip the push, post a Slack alert with the findings, and mark the task as requiring human review. Store the scan results in the task's outcome metadata for the dashboard. This provides defense-in-depth beyond `--dangerously-skip-permissions` by catching dangerous outputs before they leave the local machine. The scan runs on the git diff, so it's fast (no full-repo analysis) and only checks Claude's changes.

### Configuration change Slack notifications (#668)

When any mutating dashboard action occurs — config changes (`POST /config`), job pause/resume (`POST /pause/:job`), item skip/unskip/prioritize/deprioritize (`POST /queue/*`), PR merges (`POST /queue/merge`), task cancellations (`POST /cancel`), and WhatsApp unpair — send a Slack notification with: action type, target (job name, repo/number, or config field), old and new values (for config changes), and client IP (from `X-Forwarded-For` or socket address). Use the existing `slack.ts` webhook infrastructure with a distinct message format (no error emoji, use an info-style block). This provides a lightweight audit trail without requiring a full audit log database table (already suggested separately). The operator gets immediate visibility into operational changes, which is especially valuable when Claws is accessed from multiple devices or when debugging 'who paused ci-fixer at 2 AM?'. Include a config flag (`notifyConfigChanges`, default true) to disable if too noisy.
