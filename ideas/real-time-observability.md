# Real-time observability

### Structured task outcome summaries on log detail page (#660)

Extend the task record with structured outcome metadata captured at task completion. After a Claude task finishes (whether in issue-worker, ci-fixer, review-addresser, or improvement-identifier), parse the git state to extract: number of commits made, files changed count, insertions/deletions, PR number created or updated, and the final issue/PR state. Store this as a JSON blob in a new `outcome` column on the tasks table. On the log detail page (`GET /logs/:runId`), render this as a summary card at the top showing the concrete results — '3 commits, 5 files changed (+127/-42), PR #185 created'. For failed tasks, include the failure category (timeout, Claude error, git conflict, push rejection). This gives the operator instant visibility into what each task accomplished without scrolling through logs, and enables future analytics queries like 'average files changed per issue-worker task'.

### Job topology and pipeline visualization page (#661)

Add a `GET /topology` dashboard page that renders the Claws job pipeline as a visual flow diagram. Show the lifecycle: issues arrive → issue-dispatcher classifies → planner creates plan → implementer creates PR → pr-dispatcher classifies → ci-fixer/review-addresser/reviewer/merger process. Display each node with its current state (running with item number, idle with next-run countdown, paused, or errored with last error). Draw edges showing data flow — e.g., an arrow from 'planner' to 'implementer' labeled 'Refined label'. Highlight bottlenecks: if the Claude queue has 3 pending tasks, show which pipeline stage they're waiting at. Implement as server-rendered SVG (matching the existing no-external-dependency pattern) with auto-refresh every 30 seconds. This gives the operator a systems-level view of where work is flowing and where it's stuck, complementing the flat job list on the main dashboard.

### Queue position and estimated processing time (#662)

On the dashboard queue page (`GET /queue`), show each item's position in the Claude processing pipeline and an estimated time to start. When an item is waiting in the Claude queue, display its position (e.g., '3rd in queue') and an ETA based on the average task duration for that job type from the last 20 completed tasks in the SQLite history. For items not yet queued (discovered but waiting for the next job tick), show the job's next-run countdown from scheduler info. On the main dashboard, add a 'Queue depth' indicator showing total items across all categories with a sparkline of the last 24 hours (sampled hourly via a lightweight periodic snapshot to a new `queue_snapshots` table). This helps the operator understand throughput and make decisions like 'should I manually trigger pr-dispatcher now or wait?' Currently the queue page shows what's waiting but gives no sense of when items will be processed.

### Per-repo activity dashboard page (#733)

Add a GET /repos/:owner/:name dashboard page that shows a unified view of a single repository's Claws activity. Display: (1) pipeline status — open issues in each stage (needs refinement, refined/queued, in progress, PR open, ready for review) as a horizontal flow diagram, (2) recent task history for this repo (last 20 tasks with job name, status, duration, and outcome summary), (3) active worktrees currently checked out for this repo, (4) task success rate over the last 30 days as a simple bar chart (one bar per day, colored by success/failure ratio), (5) open PRs created by Claws with their CI and review status, and (6) scanner findings (open alert issues). Currently the operator must mentally filter the main dashboard, queue page, and logs page to understand a single repo's state — for an operator managing 10+ repos, this is significant cognitive overhead. The queue page groups by category (needs-refinement, refined, etc.) across all repos; the logs page lists all jobs chronologically. Neither gives a repo-centric view. Implementation: add a new page generator in src/pages/ that queries the tasks table filtered by repo, fetches open issues/PRs via gh CLI (cached), and renders server-side HTML matching the existing dashboard style. Add repo links from the queue page items and log entries. The repo list for the nav could come from the existing getClonedRepos() helper.
