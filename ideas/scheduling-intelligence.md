# Scheduling intelligence

### Adaptive polling interval based on work yield (#663)

Track each polling job's 'yield rate' — the percentage of ticks that actually find and process work items vs. ticks that find nothing. Store a rolling window of the last 20 ticks per job in memory (not persisted — resets on restart). When a job's yield drops below a configurable threshold (e.g., fewer than 1 in 10 ticks finds work), automatically double its interval up to a configurable maximum (e.g., 4x base interval). When work is found, immediately reset to the base interval. This is most valuable for issue-dispatcher and pr-dispatcher during quiet periods — if no new issues or PR events have occurred, polling every 5 minutes wastes GitHub API quota and log noise. During active development (many issues filed, PRs opened), the base interval is maintained. Log interval changes at info level. The base interval remains configurable via config.json and the dashboard; adaptive tuning only adjusts upward from the base. A dashboard indicator shows when a job is in 'backoff' mode.

### Immediate downstream job triggering after task completion (#664)

Add an optional `triggers` field to the Job interface that declares downstream jobs to fire when the current job's run produces actionable output. For example: when issue-dispatcher's planner posts a plan and adds the Ready label, it could record that a plan was posted. If the operator then labels the issue Refined, issue-dispatcher should ideally process it immediately rather than waiting up to 5 minutes. More concretely: when ci-fixer pushes a fix and CI passes, pr-dispatcher should immediately check for merge eligibility rather than waiting for its next tick. Implementation: after a job run completes, check if it produced work (non-empty task list in the run) and if so, call `scheduler.triggerJob()` for each declared downstream job. This preserves the existing polling as a fallback but reduces end-to-end pipeline latency from ~15 minutes (3 polling cycles) to near-instant for the common case. Add a 10-second delay before triggering to allow GitHub webhooks and CI status to propagate.

### Scheduled job overlap prevention with duration-aware staggering (#665)

Three scanner jobs (ubuntu-latest-scanner, concurrency-scanner, migration-scanner) all run at 6 AM and all iterate over every cloned repo. When they run simultaneously, they compete for disk I/O on the same repo directories and create redundant `git fetch` calls (each calls `ensureClone` which fetches). Add automatic stagger detection to the scheduler: when multiple jobs share the same `scheduledHour`, offset each by the previous job's average duration (from the last 5 completed runs in the DB) plus a 1-minute buffer. For jobs with no history yet, use a 5-minute default offset. The first job at that hour runs on time; subsequent jobs are delayed. This is transparent — the dashboard shows the effective scheduled time including the offset. Unlike 'time-of-day aware scheduling' (which suppresses jobs during work hours), this prevents simultaneous resource contention between jobs that happen to share a schedule slot.

### Scanner git operation consolidation pass (#734)

The scanner jobs (ubuntu-latest-scanner, concurrency-scanner, migration-scanner, and future scanners like secrets-scanner and actions-pinning-scanner) all iterate over the same set of cloned repos and each independently calls ensureClone() which performs git fetch. When 3-5 scanners run in sequence during scanner-dispatcher, the same repos are fetched 3-5 times within minutes. Add a pre-scan consolidation phase to scanner-dispatcher that performs a single git fetch for all repos before dispatching individual scanners. Track which repos were successfully fetched and their HEAD commit SHA in a transient Map. Individual scanners receive this pre-fetched state and skip their own fetch calls, operating on the already-current repo state. If a scanner is run outside of scanner-dispatcher (e.g., manually triggered), it falls back to its own fetch. Implementation: add a refreshAllRepos() function that iterates getClonedRepos(), runs git fetch --prune for each, and returns a Map<repoName, {fetchedAt, headSha}>. Pass this map to each scanner's run function. In ensureClone(), accept an optional lastFetchedAt parameter — if the repo was fetched within the last 30 minutes, skip the fetch. This also benefits non-scanner jobs: if issue-dispatcher runs shortly after scanner-dispatcher, it can check the same cache and skip redundant fetches. Store the fetch cache in a module-level Map with TTL expiry.
